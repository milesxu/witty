# Gateway Security Hardening

Updated: 2026-05-29

## Purpose

The native gateway now has first-pass safety checks so the PTY-backed browser
bridge is not accidentally exposed as an unauthenticated shell server.

This is still a local development gateway, not a production remote-access
service.

## Bind Policy

`witty-gateway` now parses `--bind` as an IP socket address and rejects
non-loopback addresses by default.

Allowed by default:

```text
127.0.0.1:8788
[::1]:8788
```

Rejected by default:

```text
0.0.0.0:8788
192.168.1.10:8788
```

To bind outside loopback, the caller must pass:

```text
--allow-non-loopback
```

This flag exists for explicit future experiments only; it should not be used by
the browser smoke or default local app flow.

## Origin Policy

The WebSocket upgrade now uses `tungstenite::accept_hdr` so the gateway can
inspect browser request headers before accepting the socket.

If no explicit origins are configured, the gateway accepts only loopback browser
origins:

```text
http://127.0.0.1:<port>
https://127.0.0.1:<port>
http://localhost:<port>
https://localhost:<port>
http://[::1]:<port>
https://[::1]:<port>
```

Callers can provide exact origins with repeated:

```text
--allow-origin http://127.0.0.1:8787
```

Missing or non-loopback `Origin` headers are rejected.

## Token Policy

`--token <value>` enables a simple URL query token check during WebSocket
upgrade:

```text
ws://127.0.0.1:8788/witty?token=<value>
```

If the token is configured and missing or incorrect, the upgrade is rejected
before a PTY is spawned.

The browser smoke now runs Rust gateway mode with:

```text
--allow-origin http://127.0.0.1:8787 --token witty-smoke-token
```

and connects to:

```text
ws://127.0.0.1:8788/witty?token=witty-smoke-token
```

## Remaining Gaps

- token generation is caller-owned, not generated by `witty-gateway`
- token comparison is simple string equality
- frame size limits and output backpressure are covered by
  `gateway-protocol-backpressure.md`
- no TLS support
- no reconnect/resume policy
- no multi-session authentication model

The next security milestone should focus on launch-time token generation and the
desktop-to-browser handoff flow.
